00◈ Résumé exécutif

Ce document n'est pas une roadmap — c'est un contrat Il fixe les invariants d'architecture que tout développement futur doit respecter. Il ne décrit pas comment implémenter : il dit ce qui ne doit jamais être violé, quelles que soient les évolutions du projet.
Ce qu'est LocalCMS
Position du projet
Un cockpit local de configuration, génération de fichiers, visualisation et pilotage. Prévu pour un usage public. Compatible avec des profils d'utilisateurs différents, sur des machines différentes, dans des contextes d'exécution différents.
Usage public Multi-machine Multi-usager Linux + Windows
Ce que P0 interdit définitivement
Invariants négatifs
Tout code du Core ou des modules qui suppose un username particulier, un hostname particulier, un chemin absolu local, un OS unique, une configuration personnelle hardcodée, ou un setup personnel comme condition d'exécution.
pas de /home/ghost pas de C:\Users\ghost pas de localhost hardcodé pas d'OS implicite
Ce que P0 garantit
Invariants positifs
Toute valeur locale est un profil utilisateur, pas une constante du Core. Tout comportement dépendant de l'environnement passe par $OS. Tout chemin passe par $PATH. Toute donnée utilisateur passe par $USER. Fallbacks propres dans tous les cas non configurés.
Ce que P0 ne fait pas
Périmètre explicite
P0 ne produit pas de code. Il ne liste pas d'implémentations. Il ne fixe pas de dates. Il définit les règles auxquelles tout code futur devra répondre — y compris $FORMS (M-1.1), premier livrable de P1.

—⚡ Pourquoi P0 avant M-1.1 ($FORMS)

Sans P0, $FORMS sera une dette dès sa première ligne Si $FORMS est écrit sans contrat de compatibilité, il risque d'hériter des hypothèses actuelles de localcms-v5.html : OS implicite, chemins relatifs supposés, valeurs hardcodées. Une fois $FORMS commité et utilisé par 20 modules, corriger ces hypothèses coûte un refactor global.
Sans P0 Avec P0
$FORMS génère path.join('/home/user', ...) $FORMS délègue à $PATH.resolve('@data', ...)
$VALID valide "localhost" comme hostname valide par défaut $VALID valide la forme, pas la valeur — l'usager fournit son hostname
Les modules assument bash disponible Les modules déclarent leur shell via runtime.shell, $OS résout
Les profils machines sont des constantes dans ENV Les profils machines sont dans $USER, 0 dans le Core
CONFIG/USE/DEV sont 3 sidebars séparées avec état propre CONFIG/USE/DEV sont 3 projections du même registre $REG
Pas de séparation sécurisée entre config et exécution Config ≠ exécution. Une config générée n'est jamais exécutée sans action explicite

01🔲 Compatibility Contract

Ce contrat définit les règles d'architecture que tous les composants Core et tous les modules doivent respecter sans exception. Il est non-négociable. Une règle MUST NOT n'a pas de cas particulier.

OS & Environnement d'exécution

Principe Le Core ne suppose aucun OS. Toute divergence Linux / Windows est encapsulée dans $OS et nulle part ailleurs. Les modules déclarent leurs besoins d'exécution comme métadonnées — ils n'implémentent pas la logique OS.
✓
MUST : Toute commande shell produite par un module est préfixée du shell cible. Le shell est résolu par $OS.shell() au moment de l'exécution, pas au moment de la génération.
✓
MUST : $OS est la seule source d'information sur l'OS courant. Il expose : $OS.type() → 'linux'|'windows'|'macos', $OS.sep() → séparateur de chemin, $OS.shell() → shell par défaut.
⛔
MUST NOT : Aucun module ne contient de branche if (windows) ou if (linux) dans sa logique métier. Les adaptateurs OS sont dans des adaptateurs dédiés, appelés par le Core.
⛔
MUST NOT : Aucune commande shell dans un module ne contient un chemin absolu (/usr/bin/, C:\Program Files\…). Les binaires sont résolus via des alias ou des adaptateurs.
◎
SHOULD : Les modules exposent un champ os_compat: ['linux','windows','macos','all'] dans leur manifeste. Le Core peut filtrer les modules selon l'OS détecté.

Chemins & fichiers de configuration

⛔
MUST NOT : Aucun chemin absolu dans aucun fichier Core ou module. Jamais de /home/ghost/, C:\Users\ghost\, ./opt-trading/ comme constante.
⛔
MUST NOT : Aucun username, hostname, domaine ou adresse IP dans le code source des modules. Ces valeurs appartiennent à $USER.
✓
MUST : Tous les chemins passent par $PATH avec des alias. Exemples : $PATH.resolve('@data'), $PATH.resolve('@scripts'), $PATH.resolve('@logs'). Les alias sont définis par l'usager dans data/$paths.json.
✓
MUST : Les chemins relatifs dans les templates de config générés (nginx.conf, .service, .env…) utilisent des variables {{var}} substituées à la génération par $TPL.
✓
MUST : Les fichiers de configuration LocalCMS (modules.json, $user.json…) ont tous un emplacement défini dans data/, jamais dans le répertoire de travail courant ou dans un chemin OS supposé.

Multi-machine & multi-usager

Modèle de profil Un "usager" n'est pas un utilisateur système. C'est un profil LocalCMS. Plusieurs profils peuvent coexister sur la même machine. Un profil contient : machines cibles, credentials, préférences UI, variables d'environnement personnelles. Le Core ne connaît aucun profil — il lit $USER.current().
✓
MUST : Le Core ne connaît aucune machine en dur. Les machines cibles sont dans $USER.machines[]. Les modules font référence à une machine par son id de profil, pas par son IP ou son hostname.
✓
MUST : Tout formulaire qui cible une machine expose un sélecteur de profil machine. La valeur par défaut est localhost (profil auto-créé, jamais hardcodé).
⛔
MUST NOT : Les champs user, host, port, password, ssh_key ne doivent avoir aucune valeur non vide dans le code source des modules. La valeur initiale est vide ou issue de $USER.defaults().
✓
MUST : L'interface supporte plusieurs instances simultanées de LocalCMS (via profils) sans conflit de $STORE. Chaque profil a un namespace de stockage isolé : lcms__{profileId}__*.

Fallbacks & comportements explicites

✓
MUST : Tout module dont la machine cible n'est pas configurée affiche un état not_configured visible, pas une erreur silencieuse. Le BUS émet module:not_configured.
✓
MUST : $STORE dispose d'un fallback vers sessionStorage, puis vers un store in-memory, puis vers un état non persistant signalé visuellement. L'ordre de fallback est documenté et testé.
✓
MUST : Si $OS ne peut pas détecter l'OS, il expose $OS.type() === 'unknown'. Les modules en mode unknown proposent uniquement des sorties génériques (templates de texte), jamais de commandes OS-spécifiques.
⛔
MUST NOT : Aucun comportement silencieux. Toute valeur manquante, tout fallback activé, toute dégradation de fonctionnalité est émis sur le BUS (log:add) avec niveau approprié (WARN ou INFO).

02⬡ Module Contract — Cible V1

Principe fondateur Un module LocalCMS V1 est un manifeste déclaratif. Il ne contient que des définitions de données. Il ne rend pas de HTML. Il ne valide pas lui-même. Il ne persiste pas lui-même. Il ne détecte pas l'OS. Il dit quoi — le Core dit comment.

Anatomie d'un module V1

Exemple de manifeste déclaratif

/* modules/backend-config.js — Exemple manifeste V1 cible */ /* Ce module NE CONTIENT PAS : HTML inline, logique OS, chemins hardcodés */ /* Il DÉCLARE uniquement : identité, champs, conditions, validateurs, sorties */ const MOD_BACKEND_CFG = { /* ── Identité ── */ id: 'backend_config', label: 'Backend Config', icon: '🔧', version: '1.0.0', group: 'backend', path: 'modules/backend-config.js', /* ── Capacités ── */ type: 'config', capabilities: ['render', 'generate'], os_compat: ['all'], interfaces: ['config'], toggle: { sidebar: true, boot: true, exec: false, config: true }, /* ── Sections & champs — données pures, sans HTML ── */ sections: [ { id: 'rest', label: 'REST / HTTP', icon: '🔗', fields: [ { id: 'be_name', type: 'text', label: 'Nom', value: '' }, { id: 'be_url', type: 'url', label: 'Base URL', value: '', placeholder: 'http://…' }, { id: 'be_auth', type: 'select', label: 'Auth', value: 'none', options: ['none', 'Bearer', 'Basic', 'API Key'] }, { id: 'be_token', type: 'password', label: 'Token', value: '' }, /* be_token visible UNIQUEMENT si be_auth ≠ 'none' — défini dans conditions */ ] } ], /* ── Conditions — délégué à $COND ── */ conditions: [ { if: 'be_auth', op: 'eq', value: 'none', then: 'hide', target: 'be_token' } ], /* ── Validateurs — délégué à $VALID ── */ validators: [ { field: 'be_url', rule: 'url', message: 'URL invalide' }, { field: 'be_name', rule: 'required', message: 'Nom requis' } ], /* ── Sorties — délégué à $FGEN ── */ outputs: [ { id: 'env', filename: 'backend.env', format: 'env' }, { id: 'json', filename: 'backend.json', format: 'json' } ], /* ── Bindings profil — $USER ── */ profile_bindings: [ { field: 'be_url', source: '$USER.backends[active].url', mode: 'prefill' } ], }; /* Le Core enregistre ce manifeste. $FORMS le rend. $COND y réagit. $VALID le contrôle. */

Ce qu'un module MUST / MUST NOT contenir

✓ MUST contenir
✓ Son identité (id, label, icon, version, group)
✓ Ses champs comme données {id, type, label, value, options}
✓ Ses conditions comme règles déclaratives
✓ Ses validateurs comme règles déclaratives
✓ Sa liste de sorties (formats, noms de fichiers)
✓ Ses bindings vers $USER
✓ Sa déclaration os_compat
✓ Sa déclaration runtime.shell si exec
✓ Ses capabilities[]
✓ Ses interfaces de visibilité (config/use/dev)
⛔ MUST NOT contenir
⛔ HTML inline généré manuellement (ni template string complexe)
⛔ Logique de branche if (os === 'windows')
⛔ Chemins absolus ou relatifs hardcodés
⛔ Usernames, hostnames, IPs, tokens, passwords
⛔ Appels directs à localStorage, sessionStorage, fetch()
⛔ Validation de champ en dehors du tableau validators
⛔ Références à d'autres modules par constante directe (MOD_X.foo())
⛔ Données de configuration personnelle d'un setup particulier
⛔ Imports de bibliothèques externes non déclarées dans le manifeste
⛔ État interne mutable non déclaré (variables globales cachées)

03▤ Contrat UI

CONFIG · USE · DEV — 3 projections d'un seul registre

Invariant fondamental CONFIG, USE et DEV ne sont pas trois systèmes séparés avec des modules différents. Ce sont trois projections du même registre de capacités $REG. Un module peut être visible dans plusieurs interfaces. Ce qui change entre les interfaces, c'est le filtre de projection, pas les données.
⚙ CONFIG
Filtre : capability = 'generate'
Projection des modules qui produisent des fichiers de configuration. Affiche les formulaires, les générateurs, les validateurs. L'usager configure — rien n'est exécuté sans action explicite.
render generate
▶ USE
Filtre : capability = 'exec'
Projection des modules qui exécutent des actions sur des systèmes cibles. Exécution de scripts, commandes réseau, déploiements, monitoring en direct. Chaque action est une opération consciente.
exec fetch
🛠 DEV
Filtre : interface includes 'dev'
Projection des modules d'outillage développeur. Éditeur, diff, terminal, scaffold, registre, debugger, ENV inspector. Accessible uniquement en mode dev activé.
render exec dev-only

Routing & panels

✓
MUST : Chaque panel correspond à un module registré. Le routing passe par BUS.emit('nav:goto', moduleId). Le routeur lit $REG pour valider que le moduleId existe et est actif.
✓
MUST : L'état du panel actif est dans $ENV.state.currentPanel. Il est lisible depuis n'importe quel module via le BUS. Il est persisté par $STORE entre sessions.
⛔
MUST NOT : Aucun module ne navigue vers un autre module directement. La navigation est toujours un événement BUS. Jamais d'appel direct MOD_X.render() depuis un autre module.
✓
MUST : Un module désactivé (toggle.sidebar = false ou status offline) n'est jamais rendu dans la sidebar. Il peut toujours être atteint programmatiquement via le BUS si nécessaire.

04💾 Contrat de persistance

Règle d'or Aucun module ne persiste directement. Toute persistance passe par $STORE. $STORE n'est pas accessible depuis les templates de fichiers générés. La persistance des préférences n'est pas la même chose que la persistance de la configuration système.

Les 4 couches de données

CoucheConstanteDurée de vieContenuExport/Import
Runtime $ENV.state Session courante uniquement Panel actif, onglet actif, filtres log, état UI transitoire Non
Store $STORE Persistant (localStorage) Derniers formulaires saisis, préférences UI, état panels, historique commandes Partiel
Profil utilisateur $USER Persistant (data/$user.json) Machines SSH, endpoints IA configurés, sources de données, credentials (jamais de secrets en clair) Oui — profil complet
Artefacts générés $FGEN Éphémère (download) ou explicite (API) Fichiers de configuration produits (.env, nginx.conf, .service…). Ne sont jamais stockés dans LocalCMS automatiquement. Téléchargement direct

Règles $STORE · $USER

⛔
MUST NOT : Stocker des secrets (tokens, passwords, clés SSH privées) dans $STORE. $STORE est en localStorage, non chiffré. Les secrets vont dans $USER.credentials avec marquage explicite de sensibilité.
✓
MUST : Chaque clé $STORE est préfixée : lcms__{profileId}__{moduleId}__{key}. Aucune collision entre profils ou modules.
✓
MUST : $USER est exportable via "Exporter profil" → JSON. L'import restaure les machines, endpoints et préférences, mais jamais les secrets en clair. Les champs sensitive: true sont exportés vides ou masqués.
✓
MUST : $ENV.state (runtime) n'est jamais sérialisé dans $STORE ni dans $USER. L'état runtime est toujours recalculé au démarrage.
⛔
MUST NOT : Les artefacts générés (fichiers .env, nginx.conf…) ne sont jamais stockés automatiquement dans LocalCMS. Seul l'usager décide où ils vont (download, API, clipboard).

05🔒 Contrat sécurité & sanitation

Sanitation des entrées

✓
MUST : Toute valeur saisie par l'usager dans un champ de formulaire est sanitisée avant d'être insérée dans un template de génération. La sanitation est faite par $VALID.sanitize(value, type), jamais inline dans le module.
⛔
MUST NOT : Aucune valeur de champ n'est insérée directement dans une commande shell via concaténation de chaîne. Les valeurs sont passées comme arguments nommés à un constructeur de commande, jamais interpolées librement.
⛔
MUST NOT : Aucun template de fichier généré n'est exécuté par LocalCMS automatiquement. La séparation générer ≠ exécuter est absolue. Un fichier .sh généré est un artefact texte, pas un script exécutable dans LocalCMS.
✓
MUST : Les champs de type url sont validés par une regex d'URL et ne peuvent pas contenir de caractères de contrôle. Les champs de type path ne peuvent pas contenir de traversal (../).

Secrets, chemins & exécution

Type de donnéeRègle
Passwords & tokens Jamais affichés en clair dans les logs. Jamais dans les URLs. Champs type:password sont masqués par défaut. Marqués sensitive:true dans le manifeste.
Clés SSH privées Jamais stockées dans $STORE ou $USER en clair. Référencées par chemin (ex: @home/.ssh/id_rsa via $PATH), jamais par contenu.
Chemins utilisateur Jamais traversés au-delà du contexte de travail déclaré. Les chemins sont validés par $PATH.validate() avant usage dans un template.
URLs externes Pas de redirection silencieuse. Les URLs sont validées (format) avant d'être insérées dans une config générée. Pas de résolution DNS dans LocalCMS côté Core.
Commandes exécutées Uniquement sur action explicite de l'usager. Aucune exécution automatique au chargement d'un module. Le résultat est dans un terminal sandbox, jamais évalué par LocalCMS.
Appels API Uniquement sur action explicite. Le résultat est affiché, jamais ré-exécuté. Les headers d'auth ne sont pas loggués.
Variables d'environnement Les variables dans les templates générés ({{DB_PASSWORD}}) sont des placeholders. La valeur réelle est saisie par l'usager à la génération et n'est pas conservée par LocalCMS.

Ce qui est garanti après P0
· Aucune donnée personnelle dans le code source
